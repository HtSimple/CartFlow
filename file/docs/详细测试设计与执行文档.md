## CartFlow 电商购物车结算模块

---

- **目标应用**：CartFlow 电商购物车结算系统
- **被测模块**：购物车结算模块（`backend/cartflow/checkout.py`）
- **测试框架**：Pytest 8.4.2 + FastAPI TestClient
- **测试日期**：2026-05-29
- **设计者**：AutoTestDesign 工具（AI 驱动）

---

## 1. 概述

本文档针对 CartFlow 电商购物车结算模块，展示从需求分析、覆盖项识别、测试策略选定到测试用例生成与执行验证的完整过程。全程遵循 ISTQB 基础级原则及 ISO/IEC/IEEE 29119-4 标准中定义的测试技术。

结算模块为核心业务模块，负责：
- 接收客户端购物车商品数据
- 校验商品合法性（单价、数量、库存）
- 计算订单金额（小计、原价、运费、最终金额）
- 响应状态管理（待结算 → 校验中 → 结算成功/失败）

## 4. 覆盖策略与方法

AutoTestDesign 工具根据每项覆盖项的特征，自动选择最合适的测试设计技术：

### 4.1 黑盒测试技术

**技术 1：等价类划分 (Equivalence Partitioning) — ISO 29119-4 标准技术**

应用于覆盖项：CI-01, CI-02, CI-03, CI-05, CI-06, CI-07, CI-08, CI-09, CI-10, CI-11, CI-17, CI-18, CI-19, CI-20, CI-21, CI-26, CI-27, CI-28, CI-29, CI-30, CI-31

以单价（unit_price）为例的等价类分析：
| 等价类 | 示例值 | 预期结果 |
|--------|--------|---------|
| 合法正价格 | 100.00 | 校验通过 |
| 零价格（非法） | 0.00 | 拒绝：商品单价非法 |
| 负价格（非法） | -5.00 | 拒绝：商品单价非法 |
| 非数字 | "abc" | 拒绝：商品单价非法 |
| 布尔值 | true | 拒绝：商品单价非法 |

以购买数量（quantity）为例的等价类分析：
| 等价类 | 示例值 | 预期结果 |
|--------|--------|---------|
| 合法正整数 | 2 | 校验通过 |
| 小数（非法） | 2.5 | 拒绝：购买数量非法 |
| 零（非法） | 0 | 拒绝：购买数量非法 |
| 负整数（非法） | -1 | 拒绝：购买数量非法 |
| 非数字 | "abc" | 拒绝：购买数量非法 |

**技术 2：边界值分析 (Boundary Value Analysis) — ISO 29119-4 标准技术**

应用于覆盖项：CI-04, CI-12, CI-13, CI-14

运费规则边界：原价 >= 200 免运费，否则 10 元运费

```
边界值：200.00
               199.99    200.00    200.01
                │          │          │
          收运费 10   边界值上点   收运费 0
         （离点）    （边界点）   （内点+0.01）
```

数量合法性的整数边界：
- 0 是正整数与非正整数之间的边界（TC-05 通过 `_parse_positive_int` 拒绝 0）
- 对于库存校验，`quantity > stock` 是一个判定边界

**技术 3：决策表 (Decision Table) — ISO 29119-4 标准技术**

应用于覆盖项：CI-15, CI-16, CI-22

校验与计算优先级决策表：

| 条件 | 规则 1 | 规则 2 | 规则 3 | 规则 4 |
|------|--------|--------|--------|--------|
| C1: 购物车为空 | T | F | F | F |
| C2: 所有商品单价合法 | — | F | T | T |
| C3: 所有商品数量合法 | — | — | F | T |
| C4: 所有商品数量 ≤ 库存 | — | — | — | F |
| **动作** | | | | |
| A1: 返回错误 | ✓ | ✓ | ✓ | ✓ |
| A2: 计算金额 | ✗ | ✗ | ✗ | ✗ |
| A3: 保留购物车 | ✓ | ✓ | ✓ | ✓ |

校验失败时存在短路逻辑 — 按**单价 → 数量 → 库存**依次检查，首个失败即终止。

### 4.2 白盒测试技术

**技术 4：状态转换测试 (State Transition Testing)**

应用于覆盖项：CI-23, CI-24, CI-25

结算流程的状态机模型：

```
                ┌─────────────────┐
                │    待结算        │
                │  (初始态)        │
                └────────┬────────┘
                         │ 提交结算
                         ▼
                ┌─────────────────┐
                │    校验中        │
                │  (中间态)        │
                └────────┬────────┘
                        / \
                  通过  /   \ 失败
                       /     \
                      ▼       ▼
                ┌────────┐ ┌────────┐
                │结算成功│ │结算失败│
                │(终态)  │ │(终态)  │
                └────────┘ └────────┘
```

测试覆盖的全状态路径：
1. 待结算 → 校验中 → 结算成功（TC-24，合法购物车）
2. 待结算 → 校验中 → 结算失败（TC-25，非法购物车）

注意：当前同步架构中"校验中"为瞬时中间态，函数返回后已不可观测（TC-23 记录此架构限制）。

### 4.3 补充测试技术

**幂等性测试**（CI-30, CI-31）：验证纯函数 `checkout_cart()` 对相同输入返回完全相同输出——这是确定性函数的数学性质测试。

**性能测试**（CI-28, CI-29）：验证响应时间约束 < 200ms。

**安全测试**（CI-26, CI-27）：模拟客户端篡改 `final_amount` 字段，验证服务端拒绝客户端值。

## 6. 测试工具实现

### 6.1 框架选择：Pytest

**选型理由**：

| 维度 | Pytest | 替代方案比较 |
|------|--------|-------------|
| 学习曲线 | 低 — 无需样板代码，函数名即用例 | unittest 需要继承和 setUp 模板 |
| 断言机制 | `assert` 原生 Python，自动报告失败详情 | JUnit 风格需调用 `self.assertEqual()` |
| 标记系统 | `@pytest.mark.xfail` 标记已知缺陷，`strict=True` 确保修复后变为 PASS | unittest 通过 `@unittest.expectedFailure` 但灵活性较低 |
| 参数化 | `@pytest.mark.parametrize` 单用例多数据 | 需循环或子测试 |
| 夹具 | `@pytest.fixture` 支持自动清理、作用域管理 | unittest 的 setUp/tearDown 较笨重 |
| 性能 | 轻量级，测试发现速度快 | 比 unittest 快 2-5 倍 |
| 报告 | `-v` 详细输出，`--tb=short` 精简回溯，JUnit XML 导出 | 原生支持 CI 集成 |
| FastAPI 集成 | 与 `TestClient` 无缝配合 | 需要额外配置 |

对于 CartFlow 项目，Pytest 的特性特别匹配以下需求：
- **`xfail` 标记**用于 TC-26/TC-27 — 这些用例检测 REQ-11 的已知缺陷，当前预期失败，缺陷修复后自动变绿
- **纯函数测试** — `checkout_cart()` 是无状态确定性函数，Pytest 的函数式风格天然适配
- **TestClient 集成** — 一行代码即可创建 HTTP 客户端，无需单独启动服务
