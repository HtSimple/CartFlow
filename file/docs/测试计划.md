## 1. 项目范围

### 1.1 背景

CartFlow 是一个前后端分离的电商购物车结算系统：
- **后端**：Python FastAPI 提供 RESTful API，核心结算逻辑位于 `checkout_cart()` 纯函数
- **前端**：Vue 3 + Vite 构建的用户界面（本次测试主要聚焦后端结算模块）

结算模块是整个电商平台的核心业务组件，负责从接收购物车商品数据到完成金额计算的完整流程。其正确性和安全性直接影响用户交易体验和平台财务合规。

### 1.2 总体目标

本次测试活动的总体目标：

| 目标 | 描述 |
|------|------|
| **功能正确性** | 验证结算模块对所有合法/非法输入的响应符合需求规格 |
| **财务准确性** | 确保商品小计、订单原价、运费、最终金额的计算完全正确 |
| **安全性** | 确认服务端不采纳客户端篡改的金额，防止资金欺诈 |
| **可靠性** | 验证系统在相同输入下始终返回相同结果 |
| **性能基准** | 建立结算接口的响应时间基线（< 200ms） |
| **接口契约** | 验证 HTTP 接口的请求/响应结构符合前后端约定 |

### 1.3 测试范围

| 范围 | 包含 | 不包含 |
|------|------|--------|
| 功能测试 | 全部 13 项功能需求（REQ-1 ~ REQ-13） | 第三方支付网关集成 |
| 安全测试 | 客户端金额篡改防护 | XSS、SQL 注入、CSRF |
| 性能测试 | 纯函数响应时间基准 | 高并发/负载测试 |
| 接口测试 | POST /api/checkout, GET /api/health | 前端 UI 自动化 |
| 回归测试 | 全部 36 个自动化用例 | 探索性测试 |

### 5.2 团队成员职责详述

#### 测试设计工程师 (Test Designer)

- 使用 AutoTestDesign 工具导入需求规格
- 审核工具自动识别的覆盖项（CI），修订不完整或错误的项
- 为每个覆盖项选择合适的测试技术（EP/BVA/DT/STT）
- 审查工具生成的测试用例，确认其可追溯性和完整性
- 对工具输出的测试数据进行交互式验证和调整
- **核心工作流**：覆盖项审核 → 策略确认 → 用例审查 → 数据校正

#### AutoTestDesign 工具

- 解析需求文档并输出结构化覆盖项列表
- 自动计算风险评分和测试优先级
- 根据覆盖项特征推荐并应用测试技术
- 生成满足 ISO 29119-4 标准的测试用例
- 输出可导入测试管理工具的结构化工件（CSV/JSON）
- 提供覆盖度分析与改进建议

#### 测试执行工程师 (Test Executor)

- 基于测试设计工程师确认的用例开发 pytest 脚本
- 搭建测试环境（配置 Python 依赖、pytest 设置）
- 执行自动化测试套件
- 记录测试结果，区分 PASS/FAIL/XFAIL
- 提交缺陷报告（包含复现步骤、预期/实际结果）
- 执行回归测试验证缺陷修复

## 6. 选定的测试框架及其理由

### 6.1 框架选择：Pytest 8.4.2

| 对比维度 | Pytest | unittest | 选择理由 |
|---------|--------|----------|---------|
| **学习曲线** | 低 — 函数即用例，无需样板 | 中 — 需继承 TestCase | Pytest 更简洁，便于快速编写和维护 |
| **断言方式** | `assert` 原生 Python | `self.assertEqual()` 等 | Pytest 自动报告断言细节，无需调用特定方法 |
| **已知缺陷处理** | `@pytest.mark.xfail(strict=True)` | `@unittest.expectedFailure` | Pytest 的 strict 模式确保缺陷修复后测试自动变绿，而非静默通过 |
| **夹具系统** | `@pytest.fixture` 灵活的作用域管理 | setUp/tearDown 层次化 | Pytest 夹具更模块化，可自动清理 |
| **参数化** | `@pytest.mark.parametrize` 原生支持 | 需 subTest 或自写循环 | Pytest 参数化更简洁，报告更清晰 |
| **插件生态** | 丰富（覆盖、超时、并行等） | 有限 | Pytest 生态更适合扩展 |
| **CI 集成** | JUnit XML 输出，原生支持 | 也支持 | 两者均可 |
| **FastAPI 集成** | TestClient 无缝配合 | 同样支持 | 无差异 |
