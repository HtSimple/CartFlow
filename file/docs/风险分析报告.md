# 风险分析报告

## CartFlow 电商购物车结算模块

---

- **目标应用**：CartFlow 电商购物车结算系统
- **分析工具**：AutoTestDesign
- **分析日期**：2026-05-29
- **评估标准**：风险评分 = 影响程度 × 发生概率（1-10 分制）
- **合规参考**：ISO/IEC 29119-4 测试过程标准、ISTQB 风险分析准则

---

## 1. 风险分析概述

风险分析是测试计划的核心输入。本报告对 CartFlow 购物车结算模块的 13 项功能需求逐一进行**影响 × 概率**量化评分，识别高风险区域以指导测试资源的优先分配。

### 评分标准

| 风险评分 | 等级 | 含义 |
|---------|------|------|
| 1-3 | 低 | 故障影响有限，可快速恢复 |
| 4-6 | 中 | 故障影响用户体验但不涉及资金/安全 |
| 7-8 | 高 | 故障导致核心功能不可用或资金错误 |
| 9-10 | 严重 | 故障导致资金损失、安全漏洞或合规违规 |

---

## 2. 逐项风险评估

### REQ-1：基本结算功能 — 风险评分：8（高）

| 维度 | 评估 |
|------|------|
| 影响分析 | 基本结算是用户完成购买的唯一入口。结算失败直接阻断整个交易流程，导致用户放弃购买，造成直接营收损失。 |
| 发生概率 | 中 — 正常流程较稳定，但涉及金额计算可能因数据类型问题出错 |
| 风险缓解 | 使用等价类划分覆盖单个/多个商品场景；双层测试（纯函数+HTTP） |
| **判定：高优先级** | 核心业务路径，失败会完全阻塞用户转化 |

### REQ-2：输入校验 — 风险评分：9（严重）

| 维度 | 评估 |
|------|------|
| 影响分析 | 输入校验是防止无效交易和数据损坏的第一道防线。无校验或校验不足可导致非法数据进入后续计算，造成金额错误、库存状态不一致等连锁问题。 |
| 发生概率 | 高 — 用户输入（尤其是恶意输入）不可预测 |
| 风险缓解 | 等价类划分覆盖空购物车、零价格、负价格、小数数量等边界；短路逻辑确保首次错误即终止 |
| **判定：高优先级** | 数据完整性的基础防线，校验失败会级联影响所有下游计算 |

### REQ-3：错误提示展示 — 风险评分：6（中）

| 维度 | 评估 |
|------|------|
| 影响分析 | 错误提示影响用户对问题的理解和后续操作决策。提示不准确会降低可用性，但不直接导致资金损失。 |
| 发生概率 | 低 — 错误提示与校验逻辑绑定，校验正确则提示基本正确 |
| 风险缓解 | 等价类覆盖每种错误类型对应的提示文本 |
| **判定：中优先级** | 用户体验重要但非关键路径 |

### REQ-4：金额计算 — 风险评分：9（严重）

| 维度 | 评估 |
|------|------|
| 影响分析 | 小计和原价是后续运费和最终金额计算的基础。任何计算错误将级联影响最终金额，导致多收/少收。 |
| 发生概率 | 中 — Decimal 类型可避免浮点精度问题，但汇总逻辑中乘积和累加的正确性需验证 |
| 风险缓解 | 等价类划分覆盖单商品和多商品场景；金额精度确保保留两位小数 |
| **判定：高优先级** | 财务正确性的核心，直接影响用户信任和财务合规 |

### REQ-5：运费计算 — 风险评分：8（高）

| 维度 | 评估 |
|------|------|
| 影响分析 | 运费规则（>=200 免运费，<200 收 10 元）是边界值敏感的业务规则。临界点两侧 0.01 元的差异就导致不同结果，任一方向的错误都会造成多收或少收运费。 |
| 发生概率 | 中 — 边界条件容易被忽视 |
| 风险缓解 | 边界值分析覆盖精确边界点（200.00）和两侧离点（199.99, 200.01） |
| **判定：高优先级** | 边界条件典型陷阱，且直接影响最终金额 |

### REQ-6：校验与计算顺序 — 风险评分：9（严重）

| 维度 | 评估 |
|------|------|
| 影响分析 | 校验必须先于计算。若违反此顺序，非法数据可能进入计算逻辑，导致异常状态、部分计算或脏数据。 |
| 发生概率 | 中 — 短路逻辑在代码中明确实现，但多条件时优先级顺序易出错 |
| 风险缓解 | 决策表覆盖校验失败/通过两种规则，验证金额字段在失败时全部为 0 |
| **判定：高优先级** | 数据处理的顺序保证，违反即导致数据完整性问题 |

### REQ-7：最终金额精度 — 风险评分：9（严重）

| 维度 | 评估 |
|------|------|
| 影响分析 | 最终金额 = 原价 + 运费，需保留两位小数并正确四舍五入。精度错误会导致每一笔交易的金额差异。 |
| 发生概率 | 中 — Decimal 精度处理正确性依赖具体实现 |
| 风险缓解 | 验证固定金额和带小数的金额场景；确认四舍五入（ROUND_HALF_UP）行为 |
| **判定：高优先级** | 财务计算必须以精确到分为单位，误差不可容忍 |

### REQ-8：成功结算响应结构 — 风险评分：8（高）

| 维度 | 评估 |
|------|------|
| 影响分析 | 响应结构的完整性和数值正确性是前后端契约的基础。字段缺失或结构错误会导致前端崩溃或显示错误信息。 |
| 发生概率 | 低 — 静态响应结构较为稳定 |
| 风险缓解 | 验证每个必需字段的存在和数值正确性 |
| **判定：高优先级** | 接口契约保证，直接影响前后端交互 |

### REQ-9：失败时的购物车保留 — 风险评分：7（高）

| 维度 | 评估 |
|------|------|
| 影响分析 | 结算失败后购物车被清空会直接导致用户数据丢失，造成极差的用户体验和客诉。 |
| 发生概率 | 低 — 购物车保留逻辑简单明确 |
| 风险缓解 | 验证失败响应中包含原始商品数据 |
| **判定：高优先级** | 数据完整性保证，防止用户数据丢失 |

### REQ-10：状态管理 — 风险评分：8（高）

| 维度 | 评估 |
|------|------|
| 影响分析 | 状态管理确保结算流程的正确引导。状态错误可能导致用户在错误时机执行操作（如在结算成功后又重复结算）。 |
| 发生概率 | 低 — 同步架构中状态管理简单 |
| 风险缓解 | 状态转换测试覆盖正向和失败路径的完整状态流转 |
| **判定：高优先级** | 流程正确性的关键保证 |

### REQ-11：防篡改（安全）— 风险评分：10（严重）

| 维度 | 评估 |
|------|------|
| 影响分析 | **这是全部需求中风险最高的项。** 若服务端采纳客户端传入的 final_amount，攻击者可通过篡改请求中的数据以任意金额结算——例如将 1000 元的订单改为 0.01 元。这是直接的资金损失安全漏洞，可被自动化脚本批量利用。 |
| 发生概率 | 高 — 攻击者通过浏览器开发者工具或抓包工具即可篡改请求体中的金额字段 |
| 风险缓解 | 纯函数测试 + HTTP 测试双层验证服务端始终使用自身计算的金额；当前代码中存在故意引入的缺陷，已通过 xfail 标记 |
| **判定：严重优先级** | **安全漏洞，必须立即修复** |

### REQ-12：性能 — 风险评分：5（中）

| 维度 | 评估 |
|------|------|
| 影响分析 | 响应时间超过 200ms 会影响用户体验，但不导致数据错误或资金损失。 |
| 发生概率 | 低 — 纯函数计算复杂度为 O(n)，在当前无数据库/网络 IO 场景下几乎不可能超过 200ms |
| 风险缓解 | 单商品和 50 件商品两种场景的性能基准测试 |
| **判定：中优先级** | 非功能性需求，为性能基线提供参考，但当前架构下风险极小 |

### REQ-13：幂等性 — 风险评分：7（高）

| 维度 | 评估 |
|------|------|
| 影响分析 | 相同输入产生不同输出会破坏用户信任，并在重试场景下导致不可预测的行为。 |
| 发生概率 | 低 — `checkout_cart()` 是无状态纯函数，数学上保证幂等 |
| 风险缓解 | 连续三次和不同时间段两次调用的一致性验证 |
| **判定：高优先级** | 系统可靠性的基本保证 |

## 5. 风险应对策略

| 风险评分范围 | 应对策略 | 测试资源分配 |
|------------|---------|------------|
| 9-10（严重） | **必须测试** — 全面覆盖，包括负面测试和安全测试；任何变更必须回归此项 | 分配 40% 的测试资源 |
| 7-8（高） | **重点测试** — 等价类、边界值、状态转换等多种技术组合覆盖 | 分配 40% 的测试资源 |
| 4-6（中） | **抽样测试** — 关键场景覆盖，低优先级 | 分配 15% 的测试资源 |
| 1-3（低） | **冒烟测试** — 仅在最基本的级别验证 | 分配 5% 的测试资源 |

### 风险缓解行动项

1. **REQ-11 优先修复** — 当前代码含有故意缺陷（`checkout.py:78`），修复方案为删除或注释掉第 78 行的 `final_amount = client_amount` 赋值，始终使用服务端计算值
2. **输入校验增强** — 增加对 NaN、Infinity、超大数值等边缘情况的校验
3. **监控与告警** — 在生产环境中监控结算接口的异常金额请求，发现可疑模式（如 final_amount 异常值）触发告警
4. **定期回归** — 每次变更后运行全部 36 个测试用例，确保高风险项不被回归破坏

---

## 6. 结论

CartFlow 结算模块共识别出 **1 个严重风险（REQ-11，评分 10）**、**8 个高风险项（评分 7-9）**、**2 个中风险项（评分 4-6）**。测试资源应优先集中于风险评分 7 以上的需求，占总测试工作量的 80% 左右。

REQ-11（客户端金额篡改）是当前最严重的风险点，评级为最高的 10 分——因为它同时具备**高影响**（直接资金损失）和**高概率**（无需特殊工具即可利用）。建议在功能开发和测试中优先解决此问题。
