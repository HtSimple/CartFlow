# CartFlow

CartFlow 是一个电商购物车结算模块示例，包含 FastAPI 后端、Vue 3 前端和 pytest 自动化测试。

前端预置 7 种商品，库存固定展示不可编辑，用户只选择购买量后提交结算。后端根据核心结算函数 `checkout_cart` 实现结算逻辑，整体围绕下面 13 条核心需求设计。

1. 用户必须能够对购物车中的一个或多个商品进行结算。每个商品应包含商品单价、购买数量和当前库存信息。
2. 系统应验证购物车和每个商品的输入是否合法。购物车不能为空；商品单价必须大于0，购买数量必须为整数且大于0，且购买数量不能超过当前库存。
3. 若购物车为空，或任一商品存在单价非法、购买数量非法、库存不足的情况，系统应拒绝结算，并显示对应错误信息，例如“购物车不能为空”“商品单价非法”“购买数量非法”“库存不足”。
4. 系统应根据每个商品的单价和购买数量计算商品小计，并将所有商品小计相加得到订单原价。商品小计 = 商品单价 × 购买数量，订单原价 = 所有商品小计之和。
5. 系统应根据订单原价计算运费。当订单原价大于或等于200元时，系统免收运费；当订单原价小于200元时，系统收取10元运费。
6. 系统应按照“输入校验、商品小计计算、订单原价计算、运费计算、最终金额计算”的顺序完成结算。若输入校验失败，系统应立即终止结算流程并返回失败状态，不继续进行金额计算。
7. 最终应付金额 = 订单原价 + 运费。最终应付金额不能小于0，并且金额结果应保留两位小数。
8. 当所有商品信息合法、库存充足且金额计算正常时，系统应允许结算成功，并返回订单原价、运费、最终应付金额和结算成功状态。
9. 当购物车为空、商品单价非法、购买数量非法、库存不足或金额计算异常时，系统应拒绝结算，返回结算失败状态，并保留购物车内容。
10. 购物车结算过程应包含“待结算、校验中、结算成功、结算失败”四种状态。用户提交结算请求后，订单状态由“待结算”变为“校验中”；校验全部通过后变为“结算成功”；任一校验失败后变为“结算失败”。
11. 系统不得直接信任客户端传入的最终金额，最终金额必须由系统根据商品单价、购买数量、库存和运费规则重新计算。
12. 单个购物车结算请求的处理时间应少于200毫秒。
13. 对于相同输入，系统应返回稳定且一致的计算结果。

注：为了之后测试的方便，当前代码中故意保留了一个需求 11 的缺陷：后端会错误信任客户端传入的 `final_amount`，而不是始终根据商品单价、购买数量、库存和运费规则重新计算最终金额。其他 12 条需求均符合预期。

## 后端

```powershell
conda create -n cartflow python=3.11
conda activate cartflow
cd backend
pip install -r requirements.txt
uvicorn cartflow.main:app --reload --host 127.0.0.1 --port 8000
```

核心结算函数位于 `backend/cartflow/checkout.py`，可由 pytest 或其他测试工具直接调用。

## 前端

```powershell
cd frontend
npm install
npm run dev
```

前端开发服务器默认运行在 `http://127.0.0.1:5173`，并将 `/api` 代理到 `http://127.0.0.1:8000`。

## 测试

```powershell
conda activate cartflow
pytest
```

### pytest 预期结果说明

测试文件 `backend/tests/test_cartflow_aligned.py` 包含 **36 个测试用例**，覆盖所有 13 条需求及 HTTP 接口契约，由 AutoTestDesign 工具设计生成并经过人工审核对齐。

其中针对需求 11（防客户端篡改金额）的 2 个测试用例（TC-26、TC-27）使用 `@pytest.mark.xfail(strict=True)` 标记。执行全部 pytest 时预期结果：

- **34 通过**（PASSED）
- **2 预期失败**（XFAIL）— 即 TC-26、TC-27，因代码中故意保留的缺陷而失败

可单独运行需求 11 相关测试：

```powershell
pytest -k "REQ11" -vv
```

若修复需求 11，这 2 个 xfail 测试将自动变为 PASSED。若移除了 xfail 标记但缺陷未修复，它们会以 FAILED 形式呈现。
